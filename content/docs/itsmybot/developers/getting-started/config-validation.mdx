Config validation is one of the most important parts of addon development in ItsMyBot.

The framework does three things for you:

1. copies default YAML files from `resources/`
2. validates loaded YAML against a [class-validator](https://github.com/typestack/class-validator) schema
3. exposes the result through typed helpers like `getString()` and `getSubsection()`

## Where config lives

A typical addon config setup looks like this:

        </Folder>
      </Folder>
    </Folder>
  </Folder>

At runtime, user-editable files are read from:

<Files>
  <Folder name="configs" defaultOpen>
    <Folder name="MyAddon" defaultOpen>
      <File name="config.yml" />
      <Folder name="categories">
        <File name="default.yml" />

    </Folder>
  </Folder>

The default files inside `resources/` are used as the source of truth when the addon creates missing config files.

## Single config file with `createConfig()`

Use `createConfig()` when your addon has one main YAML file.

```ts
import { Addon, ConfigFile } from '@itsmybot';
import AddonConfig from './resources/config.js';

interface Configs {
  config: ConfigFile;
}

export default class MyAddon extends Addon {
  configs = {} as Configs;

  async load() {
    this.configs.config = await this.createConfig('config.yml', AddonConfig);
  }
}
```

This will:

- look for `configs/MyAddon/config.yml`
- create it from `build/addons/MyAddon/resources/config.yml` if missing
- validate it with `AddonConfig`
- return a `ConfigFile`

## Folder of config files with `createConfigSection()`

Use `createConfigSection()` when your addon has many files of the same shape.

```ts
import { Collection } from 'discord.js';
import { Addon, ConfigFile } from '@itsmybot';
import CategoryConfig from './resources/category.js';

interface Configs {
  categories: Collection<string, ConfigFile>;
}

export default class MyAddon extends Addon {
  configs = {} as Configs;

  async load() {
    this.configs.categories = await this.createConfigSection('categories', CategoryConfig);
  }
}
```

Important behavior:

- the whole default folder is copied the first time if the target folder is missing
- files whose name starts with `_` are ignored
- each `.yml` file becomes a `ConfigFile`
- the collection key is the file path without the `.yml` extension

## Writing the validator class

Validator classes use [class-validator](https://github.com/typestack/class-validator).

Basic example:

```ts
import { IsDefined, IsString, IsBoolean, IsNumber } from 'class-validator';

export default class AddonConfig {
  @IsDefined()
  @IsString()
  channel: string;

  @IsBoolean()
  enabled: boolean;

  @IsNumber()
  interval: number;
}
```

## Nested config example

If your YAML contains nested objects, use [class-transformer](https://github.com/typestack/class-transformer) and nested validation:

```ts
import { Type } from 'class-transformer';
import { IsDefined, IsString, ValidateNested } from 'class-validator';

class ChannelSettings {
  @IsDefined()
  @IsString()
  id: string;
}

export default class AddonConfig {
  @ValidateNested()
  @Type(() => ChannelSettings)
  channel: ChannelSettings;
}
```

## What happens during validation

The `ConfigFile` loader validates with these rules:

If validation fails and a default file exists, the loader attempts one automatic correction:

- missing required values are copied from the default YAML when possible

If the file is still invalid, startup fails with formatted validation errors.

Example:

```ts
const channelId = this.configs.config.getString('channel');
const enabled = this.configs.config.getBool('enabled');
const interval = this.configs.config.getNumber('interval');
```

Nested example:

```ts
const channel = this.configs.config.getSubsection('channel');
const channelId = channel.getString('id');
```

## Working with config folders

When you use `createConfigSection()`, you get a `Collection<string, ConfigFile>`.

Example:

```ts
const category = this.configs.categories.get('tickets/support');
if (!category) return;

const name = category.getString('name');
const roles = category.getStrings('roles');
```

This pattern is used by the bundled `_Example` addon.

Real-world use cases for config folders:

- ticket categories
- preset libraries
- panel definitions
- reward tables
- template-driven message packs

## Best practices

- keep validator classes next to default YAML in `resources/`
- load config inside `load()`, not `initialize()`
- use `@IsDefined()` for required settings that must exist in production
- store config references on `this.configs` with explicit typing
- use `getSubsection()` and `getSubsections()` instead of manual object traversal
- use folder configs for repeatable entities like categories, panels, presets, or templates
